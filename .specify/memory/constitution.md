# Feeder Constitution

Feeder is a Rails application that schedules and reposts external content to
the FreeFeed social network. This constitution governs how new features are
specified, planned, and implemented via Spec Kit. Existing code predates
ratification and is not retroactively required to conform — but every new
or modified change MUST comply.

## Core Principles

### I. Rails Conventions First

New features MUST follow standard Rails conventions before introducing
custom abstractions. Specifically:

- Routes MUST be resourceful. `member` and `collection` routes SHOULD be
  avoided; prefer extracting a nested resource instead.
- Individual one-off routes for single actions SHOULD be avoided when a
  resourceful mapping fits.
- Controller actions that contain no body MUST be omitted (Rails resolves
  the view directly).
- Source files MUST end with a trailing newline (Ruby, JS, CSS, HTML, ERB).

**Rationale**: Convention over configuration keeps the codebase legible to
any Rails developer and prevents bespoke patterns from proliferating. Custom
shapes MUST be justified in the plan's Complexity Tracking table.

### II. Tests Travel With Code (NON-NEGOTIABLE)

Every code change MUST be accompanied by test changes in the same commit
scope. No feature, fix, or refactor merges without:

- New or updated tests covering the change.
- A green `bin/rails test` run prior to commit.
- A clean `bin/rubocop -f github` run prior to commit.
- Migrations verified reversible (both `db:migrate` and `db:rollback`).

Test authoring rules:

- Use FactoryBot for fixtures.
- Prefer lazy memoized helpers (`def user; @user ||= create(:user); end`)
  over eager `setup` blocks.
- Name unit tests as `test "#method should ..."`.
- Select DOM elements in view/component tests via `data-key` attributes
  (e.g. `data: { key: "stats.total_feeds" }`), not CSS classes.

**Rationale**: Tests next to code prevents drift, keeps coverage honest,
and makes every commit independently revertable. The reversibility rule
exists because production migrations have rolled back before; assume they
will again.

### III. Atomic, Reviewable Commits

Each commit MUST represent one meaningful, complete change. Concretely:

- Work is split into steps before coding starts.
- Only related hunks are staged per commit (`git add -p` is the default).
- Commit subjects are imperative and ≤ 50 characters.
- Unrelated cleanup, formatting, and feature work go into separate commits.
- `git status`, `git diff`, and `git log --oneline` are reviewed before
  pushing.

Spec Kit task lists generated by `/speckit-tasks` SHOULD map roughly
one-task-to-one-commit. PR descriptions MUST use
`.github/pull_request_template.md`.

**Rationale**: Atomic commits make review tractable, bisects useful, and
reverts safe.

### IV. Approachable UI Voice

All user-facing text — views, components, flash messages, emails — MUST be
friendly without being cutesy and clear without being clinical. Specifically:

- Voice is conversational and direct; active voice preferred.
- Implementation terminology MUST NOT leak into UI copy. Banned in user-
  facing strings: "wizard", "form", "dialog", "modal", "interface",
  "screen". Describe what the user is doing instead.
- Copy is brief; long explanations get broken into digestible pieces.
- No slang, no excessive enthusiasm, no condescension.

**Rationale**: Feeder's UX should feel like a calm tool, not paperwork and
not a chatbot. Banning UI-jargon keeps copy oriented around user intent.

### V. Observable Error Handling

Handled exceptions MUST be reported through `Rails.error`, never silently
swallowed and never bubbled past a known failure boundary without context.

- For known-fallible blocks, prefer
  `Rails.error.handle(SpecificError, context: { ... }) { ... }`.
- For caught exceptions where execution continues, call
  `Rails.error.report(e, context: { ... })` with relevant identifiers
  (feed id, user id, job arguments — whatever explains the failure).
- `rescue` without either reporting or re-raising is prohibited outside
  of trivial conversion-style rescues (e.g. `rescue ArgumentError; nil`)
  and even then SHOULD be justified.

**Rationale**: A reposting service fails in obscure ways across many feeds;
unreported swallowed exceptions become invisible degradation. Structured
context turns a stack trace into something debuggable.

## Technology Constraints

The following stack is canonical. Introducing alternatives requires an
explicit Complexity Tracking entry in the feature plan.

- **Runtime**: Ruby (managed via mise; do not change the `.ruby-version`
  pin as part of a feature change). Node (managed via mise).
- **Framework**: Rails (edge). New code MUST be compatible with the
  currently pinned edge revision in `Gemfile.lock`.
- **Database**: PostgreSQL. Schema changes go through Rails migrations.
- **Background jobs**: SolidQueue. New asynchronous work MUST use a Job
  class enqueued through ActiveJob; do not introduce a second job system.
- **Frontend**: Turbo + Stimulus, Tailwind CSS, DaisyUI. New interactivity
  SHOULD be a Stimulus controller before reaching for additional JS
  frameworks.
- **Deployment**: Kamal. Production-impacting configuration MUST be
  expressed through Kamal config rather than out-of-band shell steps.
- **Gem environment**: Assume installed. Plans MUST NOT include
  `bundle install` / gem upgrade steps as part of a feature task; gem
  changes are their own commit.

## Development Workflow

The Spec Kit loop is the default for non-trivial work:

1. `/speckit.specify` — write `specs/NNN-slug/spec.md` describing the
   change in user-visible terms.
2. `/speckit.clarify` — resolve underspecified areas (optional but
   recommended for anything touching more than one layer).
3. `/speckit.plan` — produce `plan.md`, including a Constitution Check
   gate that verifies the five Core Principles above.
4. `/speckit.tasks` — generate a dependency-ordered task list aligned
   with atomic-commit discipline (Principle III).
5. `/speckit.analyze` — optional cross-artifact consistency check before
   implementation.
6. `/speckit.implement` — execute tasks; each task ends with a passing
   `bin/rails test` and `bin/rubocop -f github`, then a commit.

Out-of-band exceptions:

- Trivial fixes (typos, single-line bug fixes, dependency bumps via the
  Dependabot stream, doc tweaks) MAY skip `specify` and go straight to a
  commit on a topic branch.
- Refactors that change no observable behavior MAY skip `specify` but
  MUST still respect Principles I–III.

Pull requests MUST:

- Use `.github/pull_request_template.md`.
- Pass CI (tests, RuboCop, linters configured in the repo).
- Be reviewed against this constitution; violations either get fixed or
  documented in the plan's Complexity Tracking section with a rationale.

## Governance

This constitution supersedes ad-hoc convention. CLAUDE.md remains the
authoritative day-to-day guidance document and SHOULD stay consistent with
this file; if they conflict, the constitution wins and CLAUDE.md is updated
in the same PR.

**Amendment procedure**:

1. Open a PR that edits `.specify/memory/constitution.md`.
2. Update the Sync Impact Report comment at the top of this file.
3. Bump the version per the policy below.
4. Update any dependent templates listed under "Template alignment".
5. Update CLAUDE.md if the change affects runtime guidance.

**Versioning policy** (semantic):

- **MAJOR**: A principle is removed, or its meaning materially narrows/
  changes in a backward-incompatible way.
- **MINOR**: A new principle or section is added, or guidance materially
  expands.
- **PATCH**: Clarifications, wording, typo fixes, examples — no change in
  obligations.

**Compliance review**: Every `/speckit-plan` invocation runs the
Constitution Check gate. Every PR review MUST verify the change conforms
to the principles above or carries a justified exception in
`plan.md` → Complexity Tracking.
